Fymo is a synchronous, per-request WSGI framework, there's no built-in async runtime hiding underneath it. Jobs and broadcasts are the two escape hatches for the two things that don't fit a request/response cycle: doing work that outlives the request, and pushing state to the browser that the browser didn't ask for.

## Declaring a task

```python
# app/jobs/emails.py
from fymo.jobs import task
from app.support.mailer import send

@task
def send_welcome_email(uid: str, email: str) -> None:
    send(to=email, template="welcome", uid=uid)
```

Every non-underscore top-level function in `app/jobs/*.py` becomes a submittable task, the same directory-walking discovery that finds `app/remote/*.py` functions and `app/broadcasts/*.py` channels. `@task` doesn't change what gets discovered, an undecorated function is registered exactly the same way, for backward compatibility. What it changes is the warning: an undecorated top-level function logs a one-line deprecation-style notice suggesting `@task` be added, since a helper that happens to be public and top-level becoming submittable by accident is an easy mistake in a file meant to stay a thin registry. Real implementation belongs in `app/support/`, underscore-prefix anything in `app/jobs/*.py` that isn't meant to be a task entry point.

Task names must be unique across every `app/jobs` module. Two modules defining `send_welcome_email` is a startup error, not a silent overwrite, the same rule broadcast channel names follow.

## Submitting a job

```python
from fymo.jobs import get_job_provider
from fymo.remote import remote

@remote
def register(email: str, password: str) -> None:
    uid = create_account(email, password)
    get_job_provider().submit("send_welcome_email", uid=uid, email=email)
```

`submit()` is fire-and-forget: no return value, no blocking, the request that called it returns immediately. Nothing here tracks progress or result either, the task itself is responsible for persisting whatever outcome it needs (a database row, say), because nothing surfaces success or failure back to the code that submitted it. A poller or the task's own side effects are what the rest of the app reads from afterward.

## Choosing a job provider

A `JobProvider` contributes exactly two things the framework calls: `register_tasks()` at startup, wiring every discovered `app/jobs` function into whatever the provider uses internally, and `submit()`, which enqueues a call to run in the background. Two providers ship built in.

| Provider | Where it runs | Durability | Needs |
|---|---|---|---|
| `threaded` (default) | In-process thread pool | Dies if the process restarts mid-run | nothing extra |
| `procrastinate` | A separate `fymo jobs-worker` process | Survives a restart, scales independently of the web tier | Postgres (`DATABASE_URL`), `pip install fymo[procrastinate]` |

`threaded` wraps the same in-process `JobRunner` fymo has always had: a bounded `ThreadPoolExecutor`, three workers by default, one bad task can't take down the pool or the request that submitted it. It's the fallback when `jobs.provider` is unset in `fymo.yml`, the same role `password` plays for auth.

`procrastinate` is a Postgres-native durable queue. A submitted job is a row, so it survives a web-process restart and gets picked up by whichever `fymo jobs-worker` process is listening, possibly several of them. Reusing the app's own database is fine, procrastinate keeps its state in its own tables.

```yaml
# fymo.yml
jobs:
  provider: procrastinate
```

<Note>
Both providers route actual execution through the same lifecycle wrapper: one log line at started, succeeded, or failed, with duration. Job arguments are deliberately never logged, only the task name, status, and timing, so a job carrying sensitive input doesn't leak it into the log stream.
</Note>

## Running a worker

`threaded` has no separate worker, `submit()` already ran the task by the time it returns. `procrastinate` does: `fymo jobs-worker` starts a process that opens its own connection, registers every discovered task, and blocks, picking up jobs as they're deferred. Calling `run_worker()` on a provider that doesn't back a durable queue raises, `threaded` has nothing for it to do.

## Declaring a channel

```python
# app/broadcasts/posts.py
from typing import Literal, NotRequired, TypedDict
from app.data.db import get_db
from app.remote.posts import Comment, ReactionCounts
from fymo.remote import NotFound


class PostActivity(TypedDict):
    kind: Literal["comment_added", "reaction_updated"]
    comment: NotRequired[Comment]
    reactions: NotRequired[ReactionCounts]


def post_activity(slug: str) -> PostActivity:
    # Deny subscribing to a post that isn't there rather than silently
    # opening a channel nothing will ever publish to.
    if not get_db().fetchone("SELECT 1 FROM posts WHERE slug = ?", [slug]):
        raise NotFound(f"post '{slug}' not found")
```

A channel is one function in `app/broadcasts/*.py`, discovered the same way jobs are, non-underscore top-level functions only, names unique across modules. Three things are read off that one function: the signature is the subscribe-time arguments, the return annotation is the payload type (checked against what actually gets published, in dev mode), and the body is the subscribe-time authorization guard. It runs on every SSE subscribe attempt, inside the same request scope remote functions get, so `current_user()` and friends work normally inside it. Returning exactly `False`, or raising, rejects the subscription; anything else, including a bare `...` body, allows it.

## Publishing events

```python
def _publish_activity(slug: str, kind: str, **payload) -> None:
    # A broadcast failure must never fail the mutation that triggered it.
    try:
        from fymo.broadcast import publish
        publish("post_activity", slug=slug, data={"kind": kind, **payload})
    except Exception:
        logger.warning("post_activity broadcast failed for %s", slug, exc_info=True)
```

`publish(channel, data=None, **args)` takes two different kinds of arguments on purpose. `args` must match the channel function's own signature, they select *which* subscribers receive the event, `slug="my-post"` only reaches subscribers who opened that exact channel with that exact `slug`. `data` is *what* gets sent, JSON-encoded straight onto the wire as an SSE frame. Like job submission, publishing is fire-and-forget: no subscribers means the payload is simply dropped, not an error, which is why the blog example wraps it in a `try`/`except` rather than letting a broadcast hiccup fail the comment it's reporting on.

In dev mode, `publish()` checks `data` against the channel's declared `TypedDict` return type and logs a warning if a required key is missing or an unrecognized one shows up. It never blocks delivery, this is a development-time nudge, not a runtime contract.

## Subscribing from Svelte

```svelte
<script>
  import { subscribe } from '$broadcast/posts';
  import { onDestroy } from 'svelte';

  let unsubscribe = subscribe.post_activity({ slug: post.slug }, (data) => {
    if (data.kind === 'comment_added') comments = [data.comment, ...comments];
    if (data.kind === 'reaction_updated') reactions = data.reactions;
  });

  onDestroy(() => unsubscribe());
</script>
```

Every `app/broadcasts` module gets its own generated `.js`/`.d.ts` pair under `$broadcast/<module>`, mirroring how `$remote` codegen works for remote functions. The generated types come straight from the channel function: its parameters become the typed `args` object, its return annotation becomes the payload type passed to the callback.

```typescript
// generated: $broadcast/posts.d.ts
export interface PostActivity {
  kind: "comment_added" | "reaction_updated";
  comment?: Comment;
  reactions?: ReactionCounts;
}
export const subscribe: {
  post_activity(args: { slug: string }, onEvent: (data: PostActivity) => void): () => void;
};
```

Under the hood `subscribe.post_activity(...)` just opens an `EventSource` against `/_fymo/broadcast/posts/post_activity?slug=...` and parses each `data:` frame as JSON. `EventSource` reconnects on its own after a transient drop, that's browser behavior, not anything fymo adds. A subscription the server actively rejects (guard returned `False`, unknown channel) closes for good instead, matching the fire-and-forget contract on the publish side: the subscription is just over.

## How a subscription resolves

The server-side handler resolves `<module>/<channel>` from the discovery registry, binds the query string against the channel function's signature (a mismatch is a 422, not a crash), runs the function body as the guard (a rejection is a 403), then hashes the module, channel name, and bound arguments into a channel key. That key is what the broadcast provider actually listens on, so two subscribers calling `post_activity` with different `slug` values never see each other's events, they're on different keys entirely even though they hit the same channel function.

<Warning>
Each open SSE connection holds a thread for its lifetime. That's fine on the threaded dev server and on gunicorn's `gthread` worker class, but it will starve a sync worker pool under any real subscriber load.
</Warning>

## Broadcast transport

The default `PostgresBroadcastProvider` uses `LISTEN`/`NOTIFY`, the same primitive procrastinate itself uses for near-instant job pickup, so it needs nothing beyond the Postgres database a fymo app already has. It works across separate OS processes: a `fymo jobs-worker` process can `NOTIFY` and a web worker holding a matching `LISTEN` picks it up, which is how a background job ends up updating a page in real time.

Postgres caps a `NOTIFY` payload at 8000 bytes, and the provider enforces that loudly at publish time rather than truncating silently. Broadcasts are meant to carry ids and small state, publish an id and let the subscriber fetch the rest, not the blob itself.
