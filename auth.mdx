```yaml fymo.yml
auth:
  enabled: true
```

That's the whole setup for email and password auth. It turns on a SQLite-backed user store, a signed session cookie, and a password provider that's ready to use out of the box.

That provider exposes four remote functions under `$remote/auth`:

- `signup`
- `login`
- `logout`
- `me`

Everything from here on is about swapping these defaults out, or building on top of them.

## Two independent axes

Fymo's auth design has two independent parts, and they don't have to agree with each other. One is how a session gets stored once a user is recognized. The other is how a user establishes that session in the first place.

Storage is simple: a single signed `fymo_session` cookie, set by fymo itself. Flow is pluggable. You can use password credentials, an OAuth redirect, or a hosted provider like Clerk that verifies its own token on every request instead of ever touching fymo's cookie.

A password login and a Google login both end up minting that same `fymo_session` cookie. Clerk skips the cookie entirely and proves identity from its own token on each request. Either way, `current_user()` doesn't care which axis produced the session, it just walks a resolver chain until something recognizes the request.

## The session cookie

```
fymo_session=<user_id>.<issued_at>.<epoch>.<signature>
```

The cookie carries a few standard safety flags:

- `HttpOnly`
- `SameSite=Lax`
- `Secure`, set whenever the request is HTTPS (directly, or behind a trust-proxy-aware load balancer)

By default, it expires after 7 days.

The signature is an HMAC-SHA256 hash over the user ID, issued time, and epoch, keyed by `FYMO_SECRET`. A function called `verify_session_token` rejects anything tampered with, expired, or malformed. All of those failures collapse into the same "no session" answer, so a bad cookie can't be told apart from a missing one.

The epoch field is what makes logout and password changes actually work. It's a per-user counter stored in the database. A token only authenticates if the epoch it was minted under still matches the user's current epoch.

Calling `logout()` or `set_password_hash()` bumps that counter, which invalidates every outstanding session for that user in one write. There's no server-side session table to sweep afterward.

## Reading the current user

```python
from fymo.auth import current_user, require_auth

@require_auth
def create_comment(slug: str, input: NewComment):
    author = current_user().email.split("@")[0]
    ...
```

`current_user()` returns the authenticated `User` object when it recognizes the request. It returns `None` otherwise. It works the same way inside remote functions and inside a controller's `getContext()`, since both run in the same request scope.

`@require_auth` wraps a function so a missing user short-circuits into a 401 before the function body ever runs. That saves every handler from checking `current_user() is None` by hand:

```python
def require_auth(fn):
    @functools.wraps(fn)
    def wrapper(*args, **kwargs):
        if current_user() is None:
            raise AuthRequired()
        return fn(*args, **kwargs)
    return wrapper
```

`AuthRequired` is a `RemoteError` under the hood. The remote router already knows how to turn any `RemoteError` into the error envelope the frontend expects, `{type: "error", status: 401, error: "unauthenticated"}`. So a gated remote function needs nothing else to signal that failure.

If your app uses `remote.explicit_optin`, you'll need to stack the decorators yourself. Put `@require_auth` under `@remote`, like this. Order doesn't matter for the auth check itself:

```python
@remote
@require_auth
def create_comment(slug: str, input: NewComment) -> Comment:
    ...
```

<Note>
`current_user()` only works inside a request scope. Remote functions and controllers get one automatically, so you rarely have to think about it. Call it anywhere else and it raises a `RuntimeError` instead of quietly returning `None`, so a misplaced call fails loudly rather than looking like the user is just logged out.
</Note>

## The password provider

The password provider is the default, and the only one, when you leave `auth.providers` unset. It contributes a full set of credential remote functions:

- `signup`, `login`, `logout`, and `me` for the core credential flow
- `request_email_verification` and `verify_email` for confirming an email address
- `request_password_reset` and `reset_password` for recovering an account

Passwords are hashed with scrypt, a standard-library algorithm that needs no extra dependency. Login always runs a real hash comparison, even against a decoy value when the email doesn't exist. That keeps response timing from being used to guess which accounts are registered.

From the frontend, these come through the generated `$remote/auth` client:

```ts
import { login, signup } from '$remote/auth';

await login(email, password);
await signup(email, password);
```

Signing up issues a session cookie right away. Separately, it sends a verification link through the `EmailSender` seam (`fymo.auth.email.EmailSender`).

The default `LoggingEmailSender` just logs that link, which is enough to develop against. Point the `auth.email_sender` config key at a class of your own when you're ready to send real mail.

## OAuth providers

```yaml fymo.yml
auth:
  providers:
    - google
    # or, any generic Authorization-Code IdP:
    - type: oidc
      id: auth0
      authorize_endpoint: ${AUTH0_AUTHORIZE_ENDPOINT}
      token_endpoint: ${AUTH0_TOKEN_ENDPOINT}
      userinfo_endpoint: ${AUTH0_USERINFO_ENDPOINT}
      client_id_env: AUTH0_CLIENT_ID
      client_secret_env: AUTH0_CLIENT_SECRET
```

`google` is a built-in `OAuthProvider` subclass with Google's endpoints already filled in. Setting `type: oidc` gives you the same Authorization Code plus PKCE flow, configured against any provider's endpoints instead.

That one config shape covers Auth0, Okta, GitHub, or a self-hosted identity provider, all without writing a line of Python.

Both kinds add two HTTP routes rather than remote functions:

- `/auth/<id>/start`
- `/auth/<id>/callback`

A browser redirect can't be a `$remote` call, so these live outside the usual remote function layer.

<Accordion title="How the PKCE handshake and callback actually work">
The PKCE verifier and the CSRF state value are both signed with `FYMO_SECRET`. They round-trip through the `state` query parameter, so fymo needs no server-side store to clean up, and no session between the redirect and the callback.

When the callback succeeds, the provider looks up the user in this order:

1. By external identity, first
2. By verified email, if there's no identity match
3. It only creates a new account if neither matches

Then it sets the same `fymo_session` cookie a password login would.
</Accordion>

## Clerk and hosted sessions

Clerk, and any hosted-session provider like it, doesn't fit the OAuth redirect shape. The identity provider already owns the login UI and the session, so every request just carries a JWT. `ClerkProvider` verifies that token on each request instead of minting a fymo session cookie:

```yaml fymo.yml
auth:
  providers:
    - clerk
```

That's genuinely all it takes, as long as `CLERK_ISSUER` or `PUBLIC_CLERK_PUBLISHABLE_KEY` is already in your environment. A typical Clerk app already has one of these set for its frontend, so this often just works with zero extra config.

<Accordion title="How fymo resolves your Clerk issuer and JWKS URL">
`from_config()` works out the issuer one of two ways. It reads `CLERK_ISSUER` directly when that's set, or decodes the issuer out of your publishable key otherwise (a `pk_test_...` or `pk_live_...` key base64-encodes the Frontend API domain).

From there, it derives the JWKS URL as `{issuer}/.well-known/jwks.json`, unless you've given one explicitly.

You can override any part of this directly in your config:

- `issuer`
- `jwks_url`
- `audience`
</Accordion>

Verification uses RS256 over JWKS, which needs the `pyjwt[crypto]` extra installed. That's an optional dependency, so core fymo stays lean for apps that don't use Clerk.

The first request from any Clerk user automatically provisions a local `User` row, using the same `resolve_or_create_user` path OAuth relies on. That means `current_user()` returns a real fymo user either way, no matter which provider signed them in.

## Conditional providers

```yaml fymo.yml
auth:
  providers:
    - password
    - type: clerk
      required: auto
```

`required: auto` skips a provider entirely when it isn't configured, rather than crashing your app on a missing constructor argument.

Before building the provider, the registry checks its `is_configured()` classmethod. For Clerk, that method just checks whether `CLERK_ISSUER` or the publishable key is set.

This lets Clerk stay dormant in local development with no Clerk environment variables at all. The moment real values land in a deployed environment, it switches on, with no conditional wiring needed in your app code. Any value other than the literal string `"auto"` raises a config error, rather than being silently ignored.

## Env vars in provider config

Write `${VAR}` anywhere in `fymo.yml`, and fymo resolves it to that environment variable's value when the config loads. If a required variable is missing, the build fails immediately and names the missing variable, rather than silently loading a config with the literal text `${VAR}` still in it.

Use `${VAR:-default}` to fall back to a default value, even an empty one, when the variable is unset.

This substitution runs on the raw YAML text before parsing. That means secrets for OAuth or OIDC providers can live in your environment, while `fymo.yml` itself stays free of anything sensitive:

```yaml fymo.yml
auth:
  providers:
    - type: oidc
      id: okta
      authorize_endpoint: ${OKTA_AUTHORIZE_ENDPOINT:-https://dev.okta.com/oauth2/v1/authorize}
      client_id_env: OKTA_CLIENT_ID
      client_secret_env: OKTA_CLIENT_SECRET
```

<Tip>
OAuth providers already read their client ID and secret from environment variables, named by the `client_id_env` and `client_secret_env` config keys. Those two never need `${}` interpolation themselves. Only the plainer config values, like endpoint URLs and issuers, need it.
</Tip>

## Custom stores and providers

Both `auth.user_store` and any provider's `class:` key accept a dotted Python path. It's the same escape hatch used elsewhere in your config file:

```yaml fymo.yml
auth:
  user_store: my_app.auth.PostgresUserStore
  providers:
    - class: my_app.auth.SamlProvider
```

A custom user store implements the `UserStore` protocol (`fymo.auth.store`). That's how you'd swap SQLite for Postgres, or anything else, without touching a single remote function. Signup, login, and reading the current user all go through `get_user_store()` underneath.

A custom auth provider subclasses `BaseProvider`, importable from `fymo.auth.providers.base`. It overrides only the axis it participates in:

- `remote_functions()` for a credential flow
- `http_routes()` for a redirect-based flow
- `resolve_session()` for a token-based flow

The framework never branches on provider type. It only ever calls those three hooks.
