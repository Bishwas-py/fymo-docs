```python app/remote/posts.py
from typing import TypedDict
from fymo.remote import NotFound

class Post(TypedDict):
    slug: str
    title: str
    content_html: str

def get_post(slug: str) -> Post:
    row = get_db().fetchone("SELECT * FROM posts WHERE slug = ?", [slug])
    if not row:
        raise NotFound(f"post '{slug}' not found")
    return row
```

```svelte
<script lang="ts">
  import { get_post } from '$remote/posts';
  import type { Post } from '$remote/posts';

  let post: Post = await get_post('hello-world');
</script>
```

Any function you write in `app/remote/*.py` and call from `$remote/<module>` on the client, no route file, no fetch call, no hand-written JSON schema. This is Fymo's remote functions layer, modeled directly on SvelteKit's: a Python function is the endpoint, and the client you call it through is generated for you at build time.

## What gets exposed

By default, every public, type-annotated top-level function in an `app/remote/*.py` module is callable from the browser. "Public" means the function name doesn't start with an underscore, and the same rule applies to the module: a file named `_helpers.py` is skipped entirely, and modules are only scanned for functions actually defined in them, not ones imported in from elsewhere.

Every parameter needs a type annotation. If one is missing, the build fails immediately rather than shipping a function that would fail unpredictably on the wire:

```python app/remote/bad.py
def fn(x):  # ValueError at discovery time: please annotate parameter 'x'
    return x
```

That "exposed by default" behavior is convenient for small apps, but it means a helper function sitting next to real endpoints in the same file is silently callable unless you underscore-prefix it. If you'd rather opt in explicitly, set `remote.explicit_optin` in `fymo.yml`:

```yaml fymo.yml
remote:
  explicit_optin: true
```

With that flag on, only functions decorated with `@remote` are discovered and dispatchable. Everything else in `app/remote/*.py` is treated as a private helper, whether or not it's underscored:

```python app/remote/posts.py
from fymo.remote import remote

@remote
def get_posts() -> list[PostSummary]:
    ...

def _format_summary(row):  # never reachable either way, opt-in or not
    ...

def helper_used_by_get_posts():  # NOT exposed once explicit_optin is on,
    ...                          # even without decoration
```

`@remote` itself doesn't wrap or change the function, it just stamps `__fymo_remote__ = True` on it. With `explicit_optin` off (the default), the decorator is a no-op as far as exposure goes, but it's still worth applying consistently so a module reads the same way regardless of which mode a project runs in.

<Note>
Discovery and the request router both call the same `is_exposed_remote_fn()` check, so what the generated client can reach and what the server will actually dispatch never drift apart.
</Note>

## The generated client

Running the Fymo build walks `app/remote/*.py` and, for every module, writes a matching `.js` file (the fetch wrapper) and a `.d.ts` file (the types) under `dist/client/_remote/`. `$remote/posts` resolves to that generated pair, so importing from it in a Svelte component gives you real autocomplete and real return types, not `any`:

```ts
// dist/client/_remote/posts.d.ts (generated, do not edit)
export interface PostSummary {
  slug: string;
  title: string;
  published_at: string;
}
export function get_posts(): Promise<PostSummary[]>;
export function get_post(slug: string): Promise<Post>;
```

Calling one of these functions does a `POST /_fymo/remote/<hash>/<fn>` under the hood, `<hash>` is a content hash of the source module so a stale cached client can't silently call the wrong version of a function after a deploy. Every call is a plain `fetch`, so it behaves like any other async call in your component: `await` it, wrap it in `try`/`catch`, race it, whatever you'd normally do.

Because the import path resolves to compiled JS with no server-only code in it, it's also safe to import lazily and keep out of your SSR bundle entirely:

```ts app/lib/auth.ts
const client = () => import('$remote/auth');

export async function login(email: string, password: string) {
  const u = await (await client()).login(email, password);
  user.set(u);
  return u;
}
```

Fymo also lets a controller thread a remote function through as a prop rather than importing it client-side. `getContext()` can hand `create_comment` straight to a component; the server replaces it with a marker during SSR, and the client runtime swaps that marker for a real fetch wrapper on hydration. That's covered on [the app/ directory](/app-directory) page, this page focuses on calling `$remote/*` directly.

## The wire format

Request and response bodies aren't plain JSON, they're encoded with [devalue](https://github.com/Rich-Harris/devalue), the same tagged-JSON format SvelteKit uses for its own form actions and load functions. Fymo ships a Python port that's wire-compatible with the JS package.

Plain JSON can't represent a `Date`, a `Set`, or a repeated reference without you writing custom (de)serialization for each case. devalue can, and Fymo's implementation currently round-trips:

<CardGroup cols={2}>
  <Card title="Date / datetime" icon="calendar">
    Python `date`/`datetime` values encode as `["Date", isoformat]` and decode back to real `date`/`datetime` objects, not strings.
  </Card>
  <Card title="Set / frozenset" icon="brackets-curly">
    Encoded as `["Set", ...]`, decoded back into a Python `set`.
  </Card>
  <Card title="NaN / Infinity / -0" icon="infinity">
    Sentinel-encoded (`-3`, `-4`, `-5`, `-6`) so they survive a JSON round trip instead of becoming `null` or throwing.
  </Card>
  <Card title="Shared references" icon="link">
    A value that appears twice in the same payload is encoded once and referenced by slot index the second time, cycles included.
  </Card>
</CardGroup>

`Decimal`, `UUID`, `bytes`, and `Enum` values also encode cleanly (as a number, a string, base64, and the enum's value, respectively), and Pydantic models are flattened field-by-field. On the way in, argument validation goes further than type coercion: if you type a parameter as a `TypedDict`, dataclass, `NamedTuple`, or Pydantic model, the incoming payload is validated against that shape, not just passed through.

## Error handling

A remote function that raises propagates to the client as a structured error rather than a generic 500. Fymo defines a small hierarchy for this in `fymo.remote`:

```python
from fymo.remote import RemoteError, NotFound, Unauthorized, Forbidden, Conflict

raise NotFound(f"post '{slug}' not found")   # 404, error: "not_found"
raise Unauthorized("sign in first")          # 401, error: "unauthorized"
raise Forbidden("not your comment")          # 403, error: "forbidden"
raise Conflict("slug already taken")         # 409, error: "conflict"
```

Any of these (or a custom `RemoteError(message, status=..., code=...)`) is caught by the router before it becomes a generic failure, and comes back as a JSON envelope carrying `status`, `error`, and `message`. An exception that isn't a `RemoteError` still gets caught, it just always reports `status: 500, error: "internal"`; in dev mode the response also includes the exception message and a traceback, both of which are stripped in production.

On the client, the generated wrapper throws a real `Error` for any of these, with `status`, `error`, and (for validation failures) `issues` attached, so a component can branch on it the way `Comments.svelte` in the example blog app does:

```ts
try {
  const c = await create_comment(slug, { body });
} catch (err: any) {
  if (err.status === 401) user.set(null);
  error = err.issues?.[0]?.msg ?? err.message ?? 'Submission failed';
}
```

A failed pydantic validation (bad argument shape, not a raised `RemoteError`) comes back as `status: 422, error: "validation"` with an `issues` array you can use to point at the specific field that failed.

## CSRF protection

The remote endpoint at `/_fymo/remote/<hash>/<fn>` only accepts `POST`. That alone rules out the classic cookie-riding GET (an `<img src="...">` or a top-level navigation carries cookies but no `Origin` header, so restricting to POST closes that path outright).

For POST requests, the router additionally checks that the `Origin` header, when present, matches the request's `Host`:

```python
def _origin_ok(environ: dict) -> bool:
    origin = environ.get("HTTP_ORIGIN")
    if not origin:
        return True
    host = environ.get("HTTP_HOST")
    ...
    return origin == expected
```

A cross-origin `fetch` or form POST always sends an `Origin` header, so a mismatched one is rejected with `403 cross_origin` before your function ever runs. Same-origin requests (including ones with no `Origin` header at all, which browsers omit for some same-origin cases) pass through.

<Warning>
This check runs whether or not you've enabled Fymo's auth module. It's the baseline protection for every remote function, not something you opt into per endpoint.
</Warning>

Alongside CSRF, every call carries a `fymo_uid` cookie: an HMAC-signed anonymous identity token, issued on first request and verified on every subsequent one. It's not an authentication credential by itself (a tampered or unsigned cookie is treated as absent and a fresh one is issued), but it gives you a stable, unforgeable handle for things like deduping a reaction or attributing an anonymous comment. Call `current_uid()` from inside a remote function to read it:

```python
from fymo.remote import current_uid

def toggle_reaction(slug: str, kind: ReactionKind) -> ReactionCounts:
    uid = current_uid()
    ...
```

For real authentication on top of that, see Fymo's auth module, layered independently on the same request.

<Tip>
Inside a remote function you also get `request_event()`, returning the current uid, remote address, cookies, and headers for that call. It only works while a request is actually in flight, calling it outside a remote-function request raises.
</Tip>
