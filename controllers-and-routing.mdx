```python app/controllers/posts.py
from app.remote.posts import get_post, get_comments, get_reactions

def getContext(id: str = ""):
    post = get_post(id)
    return {
        "post": post,
        "initial_comments": get_comments(id),
        "initial_reactions": get_reactions(id),
    }

def getDoc():
    return {"title": "Post"}
```

That controller pairs with `app/templates/posts/show.svelte`. The router resolves a request path to a controller and an action, imports `app/controllers/<controller>.py`, calls `getContext()` with whatever route params it accepts, and hands the result to the matching `.svelte` file as props. `getDoc()` supplies the page title and `<head>` content alongside it.

## Route matching

Routes come from `fymo.yml`. A `root` entry and a list of `resources` are the common case:

```yaml fymo.yml
routes:
  root: index.index
  resources:
    - posts
    - tags
```

`Router._expand_resources()` turns each resource name into four RESTful routes, each with a `{controller}/{action}.svelte` template:

| Path | Controller.action | Template |
|---|---|---|
| `/posts` | `posts.index` | `posts/index.svelte` |
| `/posts/:id` | `posts.show` | `posts/show.svelte` |
| `/posts/:id/edit` | `posts.edit` | `posts/edit.svelte` |
| `/posts/new` | `posts.new` | `posts/new.svelte` |

An app doesn't have to define every one of these templates, only the ones it actually renders. The blog example only ships `posts/show.svelte`, so `/posts` and the others simply aren't reachable in that app.

`Router.match()` tries an exact path match first, then patterns with `:param` segments, then falls back to convention-based routing for anything not declared explicitly: `/controller` resolves to `controller.index`, and `/controller/action` resolves to `controller.action`. This is what makes `/` route to `home.index` (or whatever `root:` points at) without a resource entry for it.

<Note>
`Router.add_route()` lets an app register a path directly instead of going through `resources`, for routes that don't fit the RESTful shape.
</Note>

## Route params

Segments written as `:id` in a route pattern are captured by name and passed to `getContext()` as keyword arguments, filtered down to whatever the function's signature actually declares:

```python app/controllers/posts.py
def getContext(id: str = ""):
    # id is the :id segment from /posts/:id, e.g. a post slug
    ...
```

Extra params in the match that `getContext()` doesn't accept are dropped rather than raising a `TypeError`. `fymo/core/ssr_controller.py`'s `load_controller_context()` does this filtering with `inspect.signature`, and it's the same helper both the full-page render and the soft-nav data endpoint call, so params behave identically on first load and on client-side navigation.

## getContext and getDoc

A controller module can export two functions, both optional. `getContext(**params)` returns a dict that becomes the template's props (`$props()` in the Svelte component). It can return values that aren't plain JSON, including remote functions imported from `app/remote/*`, which get serialized as callable markers the client resolves through `$remote`. `getDoc()` returns a dict of page metadata: `title` for the `<title>` tag, and an optional `head` dict with `meta` and `link` lists for `<meta>`/`<link>` tags.

```python app/controllers/index.py
def getContext():
    posts = get_posts()
    return {"hero": posts[0] if posts else None, "posts": posts[1:]}

def getDoc():
    return {
        "title": "Fymo Blog",
        "head": {"meta": [{"name": "description", "content": "A demo blog"}]},
    }
```

Neither function is required. A controller with no `getContext()` renders its template with no props; one with no `getDoc()` falls back to the app's configured name as the title.

When auth is enabled, both functions run inside a read-only request scope, so `current_user()` resolves the session cookie during SSR itself rather than only after the client hydrates. That scope is opened the same way for a full-page render and for a soft-nav data fetch, which is what keeps a logged-in user from flashing as logged-out on client-side navigation.

## Layouts

An optional `app/templates/_layout.svelte` wraps every route in the app, and an optional `app/templates/<resource>/_layout.svelte` wraps every route under that resource. A layout receives its page content through Svelte's `children` snippet:

```svelte app/templates/posts/_layout.svelte
<script lang="ts">
  let { children } = $props();
</script>

<p class="kicker"><a href="/">back</a></p>
{@render children()}
```

Each `_layout.svelte` can be paired with a same-path `_layout.py` controller (`app/controllers/_layout.py` for the root, `app/controllers/<resource>/_layout.py` for a resource) that exports its own `getContext()`/`getDoc()`, invoked the same way a page controller is. A route with no layout files gets flat props, exactly what its own `getContext()` returned. A route with a layout chain gets nested props instead: `{ leafProps, layoutProps: { root, resource } }`, and its head metadata is the result of merging every level's `getDoc()`, root first, then resource, then the leaf. Scalar keys like `title` are overridden by the more specific level; `head.meta` and `head.link` arrays concatenate instead, so a leaf's page-specific `<meta>` tags add to the root layout's defaults rather than replacing them.

## Soft navigation

By default, clicking between routes in a Fymo app doesn't trigger a full page reload. The client fetches `GET /_fymo/data/<path>`, which resolves the same route, runs the same controller and layout chain, and returns the next page's props, doc metadata, and bundle URLs as a devalue-encoded payload. The client swaps in the new leaf (and layout, if it changed) without tearing down the page.

<Tip>
A resource can opt out of soft navigation per route, forcing full page reloads for it instead:

```yaml fymo.yml
routes:
  resources:
    - posts
    - name: admin
      soft_nav: false
```
</Tip>

`Router.soft_nav_enabled(controller)` defaults to `True` for any controller not explicitly listed. If a stale client still hits `/_fymo/data/admin` after `soft_nav: false`, the endpoint answers with a `409 soft_nav_disabled` error instead of serving the data, since the resource is meant to always take a full reload.
